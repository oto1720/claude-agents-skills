# Code Review — 共通ルール

## レビューの目的

1. バグの早期発見
2. 設計・アーキテクチャの妥当性確認
3. セキュリティリスクの検出
4. チーム内の知識共有

## レビュアーの責務

### 必ず確認すること

- [ ] ロジックのバグ・エッジケース漏れ
- [ ] セキュリティ問題（インジェクション・認証・機密情報漏洩）
- [ ] パフォーマンス問題（N+1・メモリリーク・重い処理）
- [ ] テストが書かれているか、カバレッジが十分か
- [ ] エラーハンドリングが適切か
- [ ] 命名・コメントが適切か

### 重要度の分類

| 記法 | 意味 | 対応 |
|------|------|------|
| `🔴 [MUST]` | マージ前に必ず修正 | ブロッカー |
| `🟠 [SHOULD]` | できれば修正 | 強く推奨 |
| `🟡 [COULD]` | 改善案 | 任意 |
| `🟢 [NICE]` | 良い点 | 称賛 |
| `❓ [NIT]` | 細かい指摘 | 好みの問題 |

## レビューコメントの書き方

```
# ✅ 良いコメント
🔴 [MUST] TextEditingController が dispose されていません。
画面が破棄された後もコントローラーが存在し続け、メモリリークが発生します。

修正案:
@override
void dispose() {
  _controller.dispose();
  super.dispose();
}

参考: https://docs.flutter.dev/release/breaking-changes/text-editing-controller

# ❌ 悪いコメント
「ここ直して」
「この書き方は良くない」
```

## レビュイーの責務

- レビュー指摘には全て返答する（対応 or 議論 or 否定の理由）
- MUST 指摘は全て対応してから再レビューを依頼
- 設計変更を提案されたら議論してから判断する

## 自動レビュー（flutter-reviewer エージェント）

PR 作成時に `flutter-reviewer` を実行し、自動レビュー結果を確認してから人間のレビューを依頼する。

```bash
# 使い方
/flutter-review lib/features/user/
```
